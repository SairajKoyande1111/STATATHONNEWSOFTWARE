## 1. Overview

AES-256-GCM (Advanced Encryption Standard — 256-bit key in Galois/Counter Mode) is a symmetric **Authenticated Encryption with Associated Data (AEAD)** cipher. It provides two guarantees simultaneously:

| Guarantee | Mechanism |
|-----------|-----------|
| **Confidentiality** | CTR-mode stream cipher — plaintext XOR'd with a deterministic keystream |
| **Integrity / Authenticity** | GHASH polynomial MAC — 128-bit authentication tag detects any tampering |

**Mathematical formula displayed in the UI:**
```
C, T = AES-GCM(K, IV, P, AAD)
|K| = 256-bit,  |IV| = 96-bit,  |T| = 128-bit
```

Where:
- `K` = 256-bit symmetric key
- `IV` = 96-bit initialisation vector (nonce)
- `P` = plaintext
- `AAD` = additional authenticated data (empty in this implementation)
- `C` = ciphertext
- `T` = 128-bit authentication tag

This section **encrypts every cell** of the selected columns in the loaded dataset. The encrypted output preserves the original character classes (digits stay digits, letters stay letters) via a **Format-Preserving Encryption** layer built on top of the GCM keystream.

## 6. Encryption Logic — Step by Step

**Entry point:** `applyAES256Encryption()` in `client/src/lib/privacy/crypto.ts`

### Step 1 — Determine Columns to Encrypt
```typescript
const encCols = targetCols.length > 0 ? targetCols : allCols;
```
If the user selected specific columns in the left panel, only those are encrypted. Otherwise, every column in the dataset is encrypted (including text columns — unlike Paillier HE).

### Step 2 — Derive or Generate the 256-bit Key
```typescript
const keyHex =
  keyMode === "pbkdf2" && passphrase.trim().length > 0
    ? deriveKeyFromPassphrase(passphrase.trim(), pbkdf2Iter)   // passphrase path
    : generateRandomKey(seed);                                  // random path
```
The key is always a 64-character lowercase hex string (32 bytes = 256 bits). See §8 for key derivation details.

### Step 3 — Cap Dataset to 1,000 Rows
```typescript
const N = Math.min(data.length, 1000);
const subset = data.slice(0, N);
```
Browser performance limit. Full datasets are processed server-side in production.

### Step 4 — Pre-compute Per-Column Keystreams (Deterministic Mode Only)
```typescript
if (deterministic) {
  for (const col of encCols) {
    colKsBytes[col] = makeCellKsBytes(256, keyHex, hashColIV(keyHex, col));
  }
}
```
For each column, derive a 256-byte keystream from `hash(key, colName)`. This is the core of deterministic encryption: every cell in the same column shares the same keystream sequence, so identical values always produce identical ciphertext.

### Step 5 — Encrypt Each Cell
For each row and each column:

```typescript
if (deterministic) {
  // Check cache — same value in same column reuses result
  const cacheKey = col + "\x00" + rawVal;
  if (detCache.has(cacheKey)) {
    newRow[col] = detCache.get(cacheKey)!;
    continue;
  }
  const encVal = encryptFPECell(colKsBytes[col], rawVal);
  detCache.set(cacheKey, encVal);
  newRow[col] = encVal;
} else {
  // Non-deterministic: increment IV counter, generate fresh keystream
  ivCounter = (ivCounter + 1) >>> 0;
  const { cipherHex, ksBytes } = aesGcmEncrypt(rawVal, keyHex, ivCounter);
  newRow[col] = encryptFPECell(ksBytes, rawVal);
}
```

`encryptFPECell` applies a format-preserving cipher (see §10) using the keystream bytes. The output looks like the original in terms of character classes — digits encrypt to digits, letters to letters.

### Step 6 — Round-Trip Verification
After processing, the function independently encrypts and decrypts one sample value to verify correctness:
```typescript
const enc = aesGcmEncrypt(demoPlain, keyHex, demoIvSeed);
demoCipher = encryptFPECell(enc.ksBytes, demoPlain);
demoDecrypted = aesGcmDecrypt(enc.cipherHex, keyHex, demoIvSeed);
roundTripOk = demoDecrypted === demoPlain;
```

### Step 7 — Return PrivacyResult
The function returns a `PrivacyResult` object with:
- `processedData` — the array of rows with encrypted cell values
- `informationLoss: 1.0` — 100%, correct and expected for encryption
- `stats` — full cipher suite parameters, key hex, row count, round-trip result
- `compliancePassed: roundTripOk`
- `report` — full HTML compliance report string

## 7. Decryption Logic — Step by Step

**Entry point:** `applyAES256Decryption()` in `client/src/lib/privacy/crypto.ts`

### Step 1 — Re-derive the Same Key
Exact same key derivation logic as encryption (§6 Step 2). The key must match identically for decryption to succeed.

### Step 2 — Re-compute Per-Column Keystreams
```typescript
for (const col of encCols) {
  colKsBytes[col] = makeCellKsBytes(256, keyHex, hashColIV(keyHex, col));
}
```
Because the deterministic IV is derived from `hash(key, colName)` — not from the cell value or row position — the **decryption side can independently reconstruct the exact same keystream** without needing any IV embedded in the ciphertext. This is the key insight enabling format-preserving symmetric decryption.

### Step 3 — Reverse FPE on Each Cell
```typescript
newRow[col] = decryptFPECell(colKsBytes[col], encVal);
```
`decryptFPECell` reverses the modular addition applied during encryption. Subtracting the keystream byte (mod class size) undoes the shift applied to each character. See §10 for the exact formula.

### Step 4 — Return Result
```typescript
return {
  technique: "AES-256-GCM Decryption",
  informationLoss: 0,          // decryption restores original data
  compliancePassed: true,
  ...
};
```
`informationLoss` is 0 for decryption — the original data is restored.

## 8. Key Derivation

### Mode 1 — Random Key (`generateRandomKey`)
```typescript
function generateRandomKey(seed: number): string {
  const rng = makeKeystream(seed ^ 0xDEADBEEF);
  const bytes: string[] = [];
  for (let i = 0; i < 32; i++)
    bytes.push(Math.floor(rng() * 256).toString(16).padStart(2, "0"));
  return bytes.join("");
}
```
- Seeds the xorshift128+ PRNG with `seed XOR 0xDEADBEEF`
- Draws 32 bytes → 64-char hex key
- The same seed always produces the same key (reproducible demo)
- **Production gap:** A real CSPRNG (`crypto.getRandomValues()` in browsers or `crypto.randomBytes()` in Node.js) must be used

### Mode 2 — PBKDF2 Passphrase (`deriveKeyFromPassphrase`)
```typescript
function deriveKeyFromPassphrase(passphrase: string, iterations: number): string {
  let h = 0x5A827999;
  for (let i = 0; i < passphrase.length; i++)
    h = (Math.imul(h, 31) + passphrase.charCodeAt(i)) >>> 0;
  const rng = makeKeystream(h);
  for (let i = 0; i < Math.min(iterations, 200); i++) rng();  // stretching
  const bytes: string[] = [];
  for (let i = 0; i < 32; i++)
    bytes.push(Math.floor(rng() * 256).toString(16).padStart(2, "0"));
  return bytes.join("");
}
```
- **Step 1:** Hash the passphrase into a 32-bit seed using a polynomial accumulator (`h = h × 31 + charCode`)
- **Step 2:** Seed the xorshift128+ PRNG
- **Step 3:** Run `min(iterations, 200)` dummy RNG cycles — simulates PBKDF2's iteration-based key-stretching to slow down brute force
- **Step 4:** Draw 32 bytes for the final key
- **Production gap:** Real PBKDF2 uses `crypto.subtle.importKey` + `crypto.subtle.deriveBits` with SHA-256; NIST recommends ≥ 600,000 iterations

## 9. Keystream Generation (xorshift128+)

```typescript
function makeKeystream(seed: number) {
  let a = (seed ^ 0x9E3779B9) >>> 0 || 1;
  let b = (seed ^ 0x6C62272E) >>> 0 || 2;
  return () => {
    a ^= a << 13; a ^= a >> 17; a ^= a << 5;
    b ^= b >> 7;  b ^= b << 9;  b ^= b >> 8;
    const v = (((a + b) >>> 0) / 0x100000000);
    return v;  // float in [0, 1)
  };
}
```

This is a two-state **xorshift128+** PRNG:
- Two 32-bit state words `a` and `b`, initialised from the seed mixed with two fractional constants (`0x9E3779B9` = golden ratio, `0x6C62272E` = ln2)
- Each call applies bitwise shifts + XOR to both words; output = `(a + b) / 2^32` normalised to `[0, 1)`
- **Why this?** It is fast, fully deterministic, and produces uniform output — suitable for an educational keystream simulation
- **Production gap:** Real AES-GCM uses the AES block cipher itself in counter mode (CTR) to generate the keystream, with hardware AES-NI acceleration

## 10. Format-Preserving Encryption (FPE)

The FPE layer is what makes the output look readable rather than raw hex. It applies an **additive modular Caesar-like cipher per character class** using the keystream bytes.

### Encryption (`encryptFPECell`)

| Character Class | Shift Formula | Output Range |
|----------------|--------------|-------------|
| Digit (non-leading) | `(d + k) % 10` | 0–9 |
| First digit of multi-digit number | `((d−1) + k % 9) % 9 + 1` | 1–9 (never leading zero) |
| Uppercase letter | `(c + k) % 26` | A–Z |
| Lowercase letter | `(c + k) % 26` | a–z |
| Space / other | pass through unchanged | original character |

**Leading-digit special rule:** For a purely-numeric string of length > 1 (e.g. `"17937"`), the first character is always 1–9 in a valid integer. The formula maps `{1..9} → {1..9}` mod 9 to guarantee no leading zero appears after encryption (which would cause Excel to read `"05508"` as a 4-digit number).

### Decryption (`decryptFPECell`)
Exact mirror of encryption — subtracts instead of adds:

| Character Class | Reverse Formula |
|----------------|----------------|
| Digit (non-leading) | `(d − k + 10) % 10` |
| First digit of multi-digit | `((d−1) − k%9 + 9) % 9 + 1` |
| Uppercase | `(c − k + 26) % 26` |
| Lowercase | `(c − k + 26) % 26` |
| Other | pass through |

The `+10` / `+26` / `+9` additions before the modulo ensure the result is never negative.

## 11. Deterministic vs Non-Deterministic Mode

| Property | Deterministic ON (default) | Deterministic OFF |
|----------|--------------------------|-------------------|
| IV source | `hash(key, colName)` — same for all rows | Incrementing counter per cell |
| Same value in same column | → Same ciphertext | → Different ciphertext each time |
| Frequency preservation | Yes — useful for GROUP BY | No |
| Security | Weaker (frequency analysis possible) | Stronger (IND-CPA) |
| Decryptability | Works identically | Works — counter stored implicitly by position |
| Stat shown | `"✓ ON — identical values map to identical ciphertext"` | `"OFF — unique IV per cell position"` |

### Column IV Hash Function (`hashColIV`)
```typescript
function hashColIV(keyHex: string, colName: string): number {
  let h = parseInt(keyHex.slice(0, 8), 16) ^ 0xA5A5A5A5;
  const s = "COL\x00" + colName;
  for (let i = 0; i < s.length; i++)
    h = (Math.imul(h, 1664525) + s.charCodeAt(i) + 1013904223) >>> 0;
  return h;
}
```
- Takes the first 8 hex chars of the key (32 bits) and XORs with `0xA5A5A5A5`
- Runs a linear congruential hash over `"COL\0" + columnName`
- Produces a stable 32-bit IV seed per `(key, colName)` pair
- Completely independent of cell value or row index — this is what makes deterministic decryption possible without storing any IV alongside the data

## 12. IV / Nonce Strategy

### Low-Level IV Generation (`aesGcmEncrypt`)
```typescript
function aesGcmEncrypt(value, keyHex, ivSeed) {
  // Generate 96-bit IV (12 bytes)
  const ivRng = makeKeystream(ivSeed);
  const ivBytes = Array.from({ length: 12 }, () => Math.floor(ivRng() * 256));
  const ivHex = ivBytes.map(b => b.toString(16).padStart(2,"0")).join("");

  // Per-encryption keystream: key XOR ivSeed
  const combined = parseInt(keyHex.slice(0, 8), 16) ^ ivSeed;
  const ksRng = makeKeystream(combined);
  ...
}
```

- The IV is 12 bytes (96 bits) — NIST-recommended nonce size for GCM (avoids the 96-bit counter overflow weakness of longer nonces)
- The keystream for encrypting the plaintext is derived from `key[0..4] XOR ivSeed`
- In deterministic mode, `ivSeed = hashColIV(key, col)` — stable per column
- In non-deterministic mode, `ivSeed = ivCounter++` — unique per cell position

## 13. Authentication Tag (GHASH Simulation)

```typescript
// GHASH simulation — 128-bit authentication tag
const tagRng = makeKeystream(combined ^ 0xF0F0F0F0);
let tagAccum = 0;
cipherBytes.forEach(b => { tagAccum = (Math.imul(tagAccum, 131) + b) >>> 0; });
const tagSeed = (tagAccum ^ Math.floor(tagRng() * 0x100000000)) >>> 0;
const tagRng2 = makeKeystream(tagSeed);
const tagHex = Array.from({ length: 16 }, () =>
  Math.floor(tagRng2() * 256).toString(16).padStart(2,"0")
).join("");
```

The real AES-GCM authentication tag is a GHASH (Galois field polynomial multiplication) over `ciphertext || AAD`. This simulation approximates it:
1. Accumulate ciphertext bytes via a polynomial hash (`tagAccum = tagAccum × 131 + byte`)
2. Mix with a keystream-derived random value
3. Expand the mixed seed into 16 bytes (128 bits)

The tag is displayed in stats and reports as `"128-bit (16 bytes)"` but is not verified on decryption in this simulation. **Production AES-GCM verifies the tag before returning any plaintext — failure to verify raises an authentication error.**

## 14. Round-Trip Verification

After encryption, the function performs a self-test:

```typescript
const demoPlain = String(subset[0][demoCol] ?? "");
if (demoPlain !== "") {
  const demoIvSeed = (seed ^ 0xABCDEF02) >>> 0;
  const enc = aesGcmEncrypt(demoPlain, keyHex, demoIvSeed);
  demoCipher = encryptFPECell(enc.ksBytes, demoPlain);
  demoDecrypted = aesGcmDecrypt(enc.cipherHex, keyHex, demoIvSeed);
  roundTripOk = demoDecrypted === demoPlain;
}
```

- Encrypts the first cell of the first column using a fixed demo IV seed (`seed XOR 0xABCDEF02`)
- Decrypts the raw XOR ciphertext (not the FPE output) and checks it equals the original plaintext
- `roundTripOk` drives `compliancePassed`, the badge colour in the report, and the "Round-Trip Verified" stat line
- A failed round-trip indicates a keystream generation bug

## 20. Simulation Limitations & Production Gap

This implementation is explicitly an **educational browser-side simulation**. The following table maps each simulated component to its production counterpart:

| Component | This Simulation | Production Equivalent |
|-----------|----------------|-----------------------|
| AES cipher | xorshift128+ PRNG keystream (XOR) | `crypto.createCipheriv('aes-256-gcm')` (Node.js) or `crypto.subtle.encrypt` (Web Crypto) with hardware AES-NI |
| GCM mode | Polynomial accumulator hash for tag | Real GHASH over GF(2¹²⁸) with full IND-CPA + INT-CTXT security |
| CSPRNG | Seeded pseudo-random (reproducible) | `crypto.randomBytes(32)` (Node.js) or `crypto.getRandomValues()` |
| PBKDF2 | Max 200 iterations (browser cap) | `crypto.subtle.deriveBits` with ≥ 600,000 SHA-256 iterations + 128-bit salt |
| Key storage | Displayed in browser UI | HSM, AWS KMS, HashiCorp Vault, Azure Key Vault |
| Auth tag check | Not verified on decrypt | Tag verification mandatory before decryption |
| Row limit | 1,000 rows max | Streaming pipeline for arbitrarily large datasets |

> **⚠ Important:** Running `crypto.createCipheriv('aes-256-gcm')` in Node.js requires zero code changes to the cipher suite parameters — only the keystream generation function is replaced. The IV strategy, column selection, format-preserving layer, and key derivation architecture all carry over directly to production.
